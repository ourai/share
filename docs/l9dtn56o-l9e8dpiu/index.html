日报 20221018：让前端懂业务很荒唐

欧雷 发表于 2022-10-18 14:25:36

吃完午饭走路时突然脑子「抽风」，发了一个想法——

让前端懂业务这么荒唐的观点为什么能得到那么多人的认可和追捧？😂😂😂

按照主流观点，这句话会引发争议，但这是我结合对职业、行业的理解与期望中的未来协作模式而产生的想法。

这句话实际上包含了两个问题——到底什么是「前端」？「前端」为什么要懂业务？

  • 到底什么是「前端」？
    • 前端的工作
      • 业务无关
        • 网站开发
        • 基础设施建设
          • 组件库
          • 脚手架
          • 开发框架
        • 开发平台
      • 业务相关
        • 应用开发
    • 前端的职业
      • 纯前端
        • 前端工程师
      • 业务/产品向
        • 业务架构师
        • 业务线前端负责人
        • 全栈工程师
        • （Modern.js 所倡导的）应用开发者/产品开发者
        • 产品经理
  • 「前端」为什么要懂业务？
    • 被迫/被要求
      • 求职应聘或被安排做业务线前端负责人
      • 被动做全栈
      • 为了业绩考核/晋升答辩
    • 自发
      • 对业务领域感兴趣
      • 主动做全栈
      • 向产品方向、业务架构转型

「前端」理应是做「业务无关」工作的「前端工程师」——以此为前提，前端专注于展示与交互，代码中不应有业务语义，与前端沟通时的语言也是业务无关的，转化为与展示、交互强相关的用语，比如「展示 XX」——从前端的世界中剥离一切业务强相关的事物。

但是，应用中一定会有业务相关的事物，该如何处理？在用合适的架构和框架将业务语义的逻辑从 UI 组件中剥离出去之后，由非前端人员（通过 Handie 类的工具）完成领域模型定义、业务相关的状态控制等。

「非前端人员」是指除做「业务无关」工作的「前端工程师」之外的人——业务架构师、业务线前端负责人、全栈工程师/应用开发者/产品开发者、产品经理、后端工程师等。

我认为「让前端懂业务」很荒唐，是因为这是给前端人员强加要求，逼迫前端人员去承担一些本职工作之外的责任！这种思想和协作模式糟粕需要被摒弃，因为他们不会让人快乐工作！让前端做前端该做的事情！

不仅是前端人员，还包括设计人员，让他们去懂业务就是一种压迫行为！在我看来，前端人员与设计人员实则属于一类人——着眼于展示与交互，不需要去了解和背负业务上的事情！
